PAGE 2 — INTRODUCTION & PROBLEM STATEMENT
================================================================================

PROBLEM:
India has 40+ central and state government welfare schemes. Citizens rarely know
which schemes they qualify for due to complex and varied eligibility criteria
involving income, age, caste category, education, occupation, and state of
residence.

SOLUTION:
An AI agent that accepts a user's demographic profile and returns personalized
eligibility verdicts, confidence scores, and official application links — with
full reasoning transparency.

WHY AN AGENT?
Rule-based systems cannot adapt to missing data, partial matches, or nuanced
criteria. An agent can reason across multiple tools, handle uncertainty, and
summarize findings in plain language.


================================================================================
PAGE 3 — CLASSICAL AGENT
================================================================================

DEFINITION:
A classical agent operates on pre-defined rules and static logic. It has no
ability to learn, search the web, or reason beyond its programmed conditions.

EXAMPLE — Classical Eligibility Checker:

    IF age >= 18 AND age <= 35:
        IF income < 250000:
            IF category == "OBC":
                RETURN "Eligible for Post Matric Scholarship"
            ELSE:
                RETURN "Not Eligible"
        ELSE:
            RETURN "Income too high"
    ELSE:
        RETURN "Age not in range"

LIMITATIONS:
- Adding a new scheme = rewriting code
- Cannot handle partial eligibility or "maybe"
- No user feedback loop
- No context between sessions
- Single-step: one input → one output

PAGE 4 — MODERN AGENT ARCHITECTURE
================================================================================

DESIGN PHILOSOPHY:
The modern agent uses a fixed 7-step LLM-orchestrated pipeline. Tools handle
specific subtasks; the LLM synthesizes findings into a human-readable summary.

KEY COMPONENTS:

  Component              | Technology              | Role
  -----------------------|-------------------------|-----------------------------
  API Layer              | FastAPI                 | Accepts profile, returns results
  Agent Orchestrator     | Python + LangChain      | Runs 7-step pipeline
  Scheme Discovery       | DuckDuckGo + LLM        | Finds schemes via live search
  Eligibility Checker    | Rule Engine             | 8-criteria hard/soft verdicts
  Ranker                 | Calculator Tool         | Confidence scoring + balance
  Link Validator         | DuckDuckGo              | Finds official .gov.in URLs
  Summarizer             | GPT-4o-mini (OpenRouter)| Generates plain-language output
  Frontend               | Next.js + React         | Form + cards + trace viewer
  Memory                 | Session State           | Stores last profile for recheck

PAGE 5 — TOOL FLOW DIAGRAM
================================================================================

THE 7-STEP PIPELINE EXPLAINED:

  Step 1: Profile Understanding
          Normalize user input (age, income, state, category, etc.)

  Step 2: Scheme Discovery
          Call scheme_finder → DuckDuckGo queries → LLM extracts structured
          JSON → fallback to schemes.json if web fails

  Step 3: Eligibility Checking
          Call eligibility_checker → evaluate 8 criteria per scheme → verdict

  Step 4: Ranking
          Call calculator → compute confidence scores → sort by verdict and
          confidence

  Step 5: Follow-up Preferences
          Reorder if user provided hint text (e.g., "scholarship", "rural")

  Step 6: National/State Balancing
          Reserve top 2 slots for national schemes; cap at 5 results

  Step 7: Link Validation
          Call web_search → find official .gov.in URL for top 3 schemes

PAGE 7 — AGENT REASONING & TOOL ORCHESTRATION
================================================================================

ELIGIBILITY LOGIC (eligibility_checker.py):

The checker evaluates 8 criteria per scheme, split into two tiers:

HARD CRITERIA (any failure = "Likely Not Eligible"):
  - Age out of range
  - State not applicable
  - Income exceeds limit
  - Disability required but user has none

SOFT CRITERIA (each failure adds uncertainty):
  - Gender mismatch
  - Education below requirement
  - Category not in scheme's list
  - Occupation mismatch

VERDICT FORMULA:
  Hard failure          → "Likely Not Eligible"
  0 soft failures       → "Clearly Eligible"
  1–2 soft failures     → "Maybe Eligible"
  3+ soft failures      → "Likely Not Eligible"

  Confidence Score = matched_conditions / (matched + failed)

NATIONAL/STATE BALANCING:
  - Top 2 results always reserved for national schemes
    (states_applicable: ["All"])
  - Remaining 3 slots filled by highest-confidence schemes
  - Ensures users always see central govt schemes even if state schemes
    score higher

FOLLOW-UP PREFERENCE RERANKING:
  - User can type: "I need a scholarship" or "I am from a rural area"
  - Keywords (scholarship, rural, low income, national, etc.) boost
    matching schemes
  - Prevents one-size-fits-all ordering

PAGE 9 — ANALYSIS & REFLECTION
================================================================================

WHAT WORKED WELL:

  1. Graceful Degradation
     The agent works without an LLM (uses fallback summary) and without
     internet (uses schemes.json). This makes it robust for real-world
     deployment.

  2. Transparency
     The agent trace log shows every decision. Users and auditors can verify
     how verdicts were reached — critical for welfare applications where
     trust matters.

  3. Hard/Soft Failure Separation
     Mirrors real government logic. Some criteria are absolute (state, income
     cap), others are flexible (occupation, gender). This prevents
     over-rejection.

  4. LLM for Narration Only
     Using GPT-4o-mini only for the final summary (not for eligibility
     decisions) ensures deterministic, auditable verdicts. The LLM cannot
     hallucinate incorrect eligibility.

LIMITATIONS:

  1. Pipeline is Fixed, Not LLM-Driven
     The agent always runs all 7 steps in order. A true autonomous agent
     would let the LLM decide which tools to call and when.

  2. Session-Only Memory
     Memory is lost when the browser tab closes. Persistent memory (database
     or vector store) would allow users to track applications over time.

  3. Static Scheme Data
     schemes.json needs manual updates. Integrating a live government API or
     regular web-scraped updates would keep data fresh.

  4. No Multi-Turn Conversation
     The agent handles one profile submission at a time. A conversational
     interface could let users refine their profile across turns.

KEY LEARNING:
  Modern LLM-based agents separate reasoning (LLM) from verification
  (rules/tools). For high-stakes domains like welfare eligibility, this
  hybrid approach is safer than fully LLM-driven decisions, which can
  hallucinate or be inconsistent.

PAGE 10 — CONCLUSION & REFERENCES
================================================================================

CONCLUSION:

This project demonstrates that a modern AI agent, even one with a fixed
pipeline, significantly outperforms a classical rule-based system across all
evaluation dimensions: it handles partial eligibility, discovers schemes
dynamically, provides explainable confidence scores, balances national and
state results, and communicates findings in natural language.

The government scheme eligibility domain is an ideal use case for agentic AI —
complex criteria, large scheme space, and high user need for guidance — and
this implementation provides a production-ready foundation for real deployment.
